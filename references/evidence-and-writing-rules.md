## 1. 证据优先原则

先确定软件实际做了什么，再决定材料写什么。材料不得反向要求代码“看起来支持”某项功能。对每项候选功能回答五个问题：用户从哪里进入、输入什么、哪个接口或入口接收、核心逻辑在哪里执行、结果写到哪里或如何返回。任一关键环节断开，都不能把完整闭环写成已实现功能。

使用代码路径、符号、行号、调用关系、构建日志、冒烟测试和真实截图。README、产品设想或文件名只用于定位，不是完成证据。证据结论要能被另一名复核者从冻结源码重现。

## 2. 证据等级 0—4

| 等级 | 可见证据 | 允许写法 | 禁止写法 |
|---|---|---|---|
| 0 | 只有名称、注释、README、依赖或设计描述 | 不申报；列入排除项 | “系统支持……” |
| 1 | 只有模型、Schema、表、种子数据、未调用函数 | 最多写“预留数据结构”，通常不写 | 写成用户可用功能 |
| 2 | 后端函数/API 存在，但无可用用户路径或链路不完整 | 谨慎写内部能力或实现基础，并说明限制 | 写成完整前台闭环 |
| 3 | 可追踪的 UI/交互入口 → API/命令入口 → 业务逻辑 → 结果返回或持久化链 | 可写已实现功能 | 声称已运行验证或配真实运行图 |
| 4 | 具备 3 级证据，并通过隔离运行验证且有经复核的真实截图 | 可作为核心功能写入操作说明和申请表 | 超出实测路径泛化到未验证场景 |

同一模块可以有不同等级。例如“创建记录”为 4 级，“任意修改和删除记录”只有 1 级时，只写创建，不用“完整增删改查”概括。

## 3. 功能链审查方法

按以下顺序建立证据：

1. 找到应用入口、菜单、路由、窗口、CLI 子命令或公开接口。
2. 追踪触发事件、参数校验和权限判断。
3. 追踪网络请求、控制器、处理器或调用边界。
4. 追踪业务服务、算法、规则和状态变化。
5. 追踪数据库、文件、缓存、第三方调用或返回对象。
6. 检查结果是否真正显示给用户，异常是否被处理。
7. 在冻结 manifest 中记录每层文件、符号和行号。
8. 运行构建和主路径，记录命令、输入、输出和时间。
9. 截图并映射到功能 ID。

Web 功能的 3 级证据通常至少包含 `ui`、`api`、`service/business/persistence` 中的完整链。桌面、移动端、CLI、插件或算法库没有 HTTP API 时，用其真实交互入口和调用层替代，并在证据文件中解释，不得虚构 API。

权限和角色单独验证。数据库中存在 `admin`、`operator`、`auditor` 等值，不表示这些角色都能通过当前界面进入；必须检查登录判断、路由守卫、菜单展示和服务端授权。客户端隐藏按钮也不等于服务端禁止调用。

本地保存与服务端保存必须区分。浏览器 `localStorage`、进程内缓存、后端数据库、对象存储和外部平台是不同的数据边界；说明书要准确写明保存位置、生命周期和跨端能力。

条件降级路径要如实写。例如先调用视觉模型、失败后调用本地 OCR，应写为“按配置优先……，不可用时回退……”，不能只写一个固定实现。

## 4. 容易误判的代码信号

以下信号不能单独证明功能完成：

- 依赖清单里安装了 OCR、支付、消息、AI、图表或数据库包。
- 存在实体、表名、迁移、枚举或 DTO，但没有可达入口。
- 页面有按钮、文字或图标，但事件未绑定、接口地址不存在或返回模拟值。
- 后端有路由，但前端没有入口，或权限守卫实际阻断。
- `dist`、`build`、缓存或旧备份里存在实现，当前源码没有。
- 测试、种子数据、演示账号或 mock 响应呈现了理想流程。
- 名为 `edit/delete/export/pay/approve` 的函数未被调用。
- 数据创建接口存在，就概括为全套资源管理。
- 仅存在模型提示词，就声称模型输出质量、准确率或医学结论得到保证。

发现这些信号时，继续追踪调用或把功能降级。不要为了让说明书丰满而填补断点。

## 5. 运行与截图证据

优先采用项目已有的安装说明、锁文件和脚本；记录实际使用的运行时和依赖版本。构建前识别 `.env`、数据库、外部 API、消息服务、生产地址和迁移风险，使用本地隔离配置、临时数据库和脱敏样例。若运行会产生外部副作用，先停下并取得授权。

最低验证集：

- 前端或客户端构建成功；后端或服务可启动。
- 至少一条匿名或登录入口路径。
- 每个 4 级核心功能的一条正常路径。
- 关键校验失败或空状态的一条异常路径。
- 管理端和用户端分别验证其真实可达菜单。
- 截图时系统名称与登记名称一致。

先建立“功能 ID—页面/状态—测试输入—预期结果—截图编号”覆盖表。所有纳入说明书的 4 级功能必须有真实运行图；3 级功能至少在说明书中完整描述并说明尚未运行复核的边界。截图不仅覆盖入口，还应按项目实际覆盖输入状态、处理反馈、结果页、查询/管理页和关键异常提示。不得因为已经取得首页、仪表盘或登录页截图就停止补采。

浏览器控制采用统一自动降级链：

1. 当前环境存在可用的 Codex/ChatGPT 内置浏览器时，优先控制本地页面；不要假定 VS Code Codex 或 Codex CLI 必然具备内置浏览器。
2. 内置浏览器不可用时使用可选的 Python Playwright + Chromium；通过 `screenshot_scenarios.json` 重放真实用户路径，默认 1440×900 视口且不使用超长 `full_page` 截图。
3. Playwright 不可用，或验证码、硬件、人工登录等环节无法自动完成时，由用户人工截图并如实记录工具。
4. 仍未取得截图时生成 `manual_screenshot_checklist.md`，保留明确补图项并阻断提交。

浏览器工具失败与系统功能失败分开记录。使用用户提供的图片、视频或文件样例执行上传、检测、识别、预览等功能时，实际选择样例并等待系统结果，记录样例路径或来源、界面 URL、窗口尺寸和截图工具；不得手工把预期结果贴进页面。

截图规则：

- 使用真实运行窗口；设计图、静态 HTML、Figma 图不能冒充。
- 保持合理分辨率和清晰字号；正文嵌图后仍可辨认主要标签。
- 一个画面来自同一次渲染。允许裁边和统一缩放，不允许拼接两个状态形成不存在的界面。
- 使用专门的演示数据；不得显示真实身份证、手机号、邮箱、病历、订单、地址或人脸等个人信息。
- 不显示密码、令牌、API key、数据库连接串、测试账号清单、绝对磁盘路径、控制台堆栈、浏览器“不安全”警告或开发者工具中的敏感内容。
- 若必须遮盖，使用不可逆实色遮盖并保留操作语义；记录遮盖动作。
- OCR 或人工搜索旧名称、旧简称、错别字和版本；仅自动检查不能代替肉眼复核。
- 每张图在 `figures_manifest.json` 绑定功能 ID、捕获时间、运行 URL、视口、场景 ID、操作路径、脱敏测试数据说明、SHA-256 和复核结论。
- `capture_tool` 只如实使用 `codex_in_app_browser`、`playwright_chromium`、`user_chrome`、`windows_snipping_tool` 或 `manual_upload`。
- 自动采集只能辅助真实性检查，所有人工复核字段必须初始化为 `false`。逐图肉眼确认后才可手工设为 `true`；图片哈希改变时旧复核立即失效。

缺图时允许草稿写 `【后期补图：图题】`，但这属于明确占位，不得进入可提交版本。架构图和流程图不是运行截图，图题和清单必须区分。

## 6. 说明书写作方法

采用“事实—处理—结果—限制”的段落结构。一个小节尽量用一至三段承载完整逻辑，每段有具体主语、输入、处理顺序、状态变化和输出，不把简单句拆成大量项目符号。段落长度以能讲清一个闭环为准，不机械追求字数。

采用四章结构，且第三、四章固定在文档末尾：

1. 软件设计说明：编写目的、适用范围、总体架构、全部纳入的功能模块、数据与接口设计、权限和安全、异常与边界。
2. 软件使用说明：部署或访问方式、初始化、登录/权限，并按真实菜单覆盖每项 3—4 级用户功能的入口、前置条件、输入、处理、结果和失败反馈。
3. 软硬件运行环境：分别写硬件运行环境、软件运行环境、运行条件与环境边界，并优先用表格列明经验证的配置。
4. 知识产权声明：写声明范围、第三方软件与资料边界、使用/复制/保密要求；未知权利主体不猜写，权属以正式申请表、证明材料和签章文件为准。

等级 3—4 的已实现用户功能默认全部进入说明书。确需排除时在证据矩阵明确 `include_in_manual=false` 和原因；不能只详写一两个核心功能，把其他已实现菜单、查询、管理、设置或异常路径完全省略。

架构段落说明层之间的责任和真实调用，不罗列所有框架名。模块段落按业务能力分组，不按源码文件逐个翻译。数据段落只写确实使用的数据对象、来源、流向、保存位置和生命周期。安全段落不得声称“绝对安全”“完全保护”；写实际存在的认证、授权、脱敏、校验和日志机制。

操作说明包含入口、前置条件、输入、点击/命令、系统处理、结果和失败反馈。截图放在相应文字之后，图题与画面一致。不要写截图中没有的按钮，也不要把后台自动行为说成用户可选择项。

弱化模板化和 AI 风味：

- 用项目中的真实对象、角色、状态和流程替代“提升效率、优化体验”等空泛句。
- 避免每段都以“系统通过……实现……”开头。
- 避免“首先、其次、再次、最后”反复出现；用因果、条件和状态衔接。
- 不编造精确性能、准确率、并发量、合规认证或算法效果。
- 技术名只在解释实现边界时出现，不堆叠框架清单。
- 同一功能在设计章说明机制，在使用章说明操作，不复制同一段换词。

说明书的中文正文用宋体、各级标题用黑体、英文和数字用 Times New Roman；所有可见文字、目录、页眉页脚、图题、表题和超链接显示为纯黑 `000000`。不要使用主题色、彩色标题或彩色强调。总体架构图和流程图优先采用白底、黑字、黑色线框和灰度填充，确保打印与复印后仍可辨认。

## 7. 申请表功能技术说明

先从 3—4 级功能中提炼 4—8 个核心能力，再写技术实现。去空白字符保持 500—1300 字，并与说明书措辞和边界一致。推荐顺序：软件服务对象和输入、核心业务闭环、用户与管理功能、数据处理和存储、权限/校验/异常、部署环境和技术结构。

不要写：未实现功能、营销口号、行业排名、无法证明的 AI 准确率、绝对安全、用户数量、商业价值、真实客户名称、与著作权无关的专利式效果。不要直接复制 README 或说明书整段；申请表需要密集概括，但事实不能变化。

生成前做字符计数：`len(re.sub(r"\s+", "", text))`。程序量、语言、数据库和运行环境从冻结事实与实际构建环境取得，不按常识补齐版本。

## 8. 功能边界表述

用范围限定词消除过度承诺：

- 数据只保存在浏览器：写“在当前浏览器本地保存”，不写“同步到个人账户”。
- 后端会保存但界面开关只控制本地历史：分别写清两个行为，不概括为“不留存”。
- 只有新增和导入：写“支持新增与导入”，不写“支持统一维护或完整增删改查”。
- 仅管理员可达：写明当前可达角色；不要列出数据库中未接通的角色。
- 外部 AI 可选：写“在配置可用时调用”，并说明失败后的回退或提示。
- 只有推荐结果：不写“已完成预约、支付、排队或医院确认”。
- 只有辅助判断：不写“自动诊断”“替代专业人员”。

把不申报功能和原因写入 `excluded_claims`。这是材料可信度的组成部分，不是缺陷清单；它防止后续写作者从按钮、模型或旧截图重新引入越界内容。
